# 重要：后端代码使用说明

## 说明

现有 `CR660X/` 和 `JGC-Q10/` / `JGC-Qx/` 目录下的脚本为**原始后端代码**。

## 限制

这些脚本在编写时**完全没有考虑 UI 层**，存在以下问题：

1. **输出不规范**
   - 有些是进度输出，有些是调试输出
   - 日志格式不统一
   - 没有状态码机制

2. **错误处理不统一**
   - 有些用 exit code，有些打印错误信息
   - 没有标准化的错误分类
   - 失败原因难以程序化判断

3. **配置硬编码**
   - IP 地址写死在脚本里
   - 文件路径写死
   - 难以参数化

4. **缺乏状态反馈**
   - 无法知道具体进行到哪一步
   - 无法获取进度百分比
   - 成功/失败判断方式多样

## 用途

✅ **可以用来**：
- 理解刷机流程是如何操作的
- 学习各阶段的网络交互逻辑
- 参考认证算法和API调用方式
- 分析可能遇到的问题

❌ **不建议用来**：
- 直接作为新系统的后端代码
- 封装成函数库调用
- 基于这些脚本二次开发

## 后期计划

**后端代码会重构**，原因：

1. 需要标准化输出格式（JSON）
2. 需要统一错误处理机制
3. 需要参数化配置
4. 需要状态回调机制
5. 需要与 UI 层分离

## 参考使用

如果要理解某个流程如何实现：

**JGC 刷机流程**：
```
官方→PDCN  → 参考：official_flash_all.py
PDCN→BL   → 参考：pdcn_put_bootloader.py + pdcn_flash_bootloader.py
BL→最终固件 → 参考：uboot_flash.py
```

**CR660X 刷机流程**：
```
登录→开SSH→传文件→刷机 → 参考：test_login1.sh / unicom_flash.sh
升级模式        → 参考：update.sh
```

## 警告

**不要依赖现有脚本的输出格式**，因为重构后会完全改变。
