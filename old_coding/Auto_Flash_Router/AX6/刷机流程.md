# AX6 (RA69) 完整刷机流程 —— 官方参考文档

> **这是唯一事实来源。** 其他 AI 模型/TUI 开发者请以此文档为准。

## 前置条件

| 项目 | 说明 |
|------|------|
| 攻击机 | Alpine/Debian，能访问 192.168.31.0/24 和 192.168.1.0/24 |
| 辅助路由器 | 提供 WiFi SSID `AX6-EXPLOIT` / 密码 `12345678`，关闭 DHCP |
| exploit_server | 在辅助路由器上运行 `exploit_server.py`，监听 169.254.31.1:80 |

> **重要**: exploit_server 在 **辅助路由器上持续运行**，攻击机（Alpine）侧**不需要**启动 exploit_server。攻击机只负责调用 AX6 的 API 触发攻击链。

## AX6 硬件信息

| 项目 | 值 |
|------|-----|
| 型号 | Redmi 路由器 AX6 (RA69) |
| SoC | IPQ8071A (Qualcomm aarch64) |
| 闪存 | 256MB SPI-NAND |
| 内存 | 512MB |
| 默认 IP | 192.168.31.1 (stock) → 192.168.1.1 (OpenWRT) |
| 出厂密码 | `admin` |

## 双分区架构

```
mtd12: rootfs    (0x023c0000 ≈ 35.75MB)  — 主分区
mtd13: rootfs_1  (0x023c0000 ≈ 35.75MB)  — 备胎分区
```

- `flag_boot_rootfs=0` → 从 rootfs (mtd12) 启动
- `flag_boot_rootfs=1` → 从 rootfs_1 (mtd13) 启动
- 通过 `/proc/cmdline` 中的 `ubi.mtd=rootfs` 或 `ubi.mtd=rootfs_1` 确认当前启动分区

## 固件文件清单

| 阶段 | 文件 | 大小 | 路径 |
|------|------|------|------|
| Stage 1 | `RA69_1.0.16.bin` | ~40MB | `firmware/` (降级固件，仅在版本 >1.0.x 时需要) |
| Stage 2 | `xiaomimtd12.bin` | 35.8MB | `files/openwrt/` (过渡 OpenWRT 原始镜像) |
| Stage 3 | `ax6-mibib-stock.bin` | 256KB | `files/openwrt/` (自定义分区表) |
| Stage 3 | `ax6-uboot-stock.bin` | 654KB | `files/openwrt/` (pepe2k u-boot_mod) |
| Stage 4 | `immortalwrt-*-factory.ubi` | 15.9MB | `firmware/` 或 `files/xiaobai/` (最终固件) |
| Stage 5 | `overlay-new.tar.gz` | 7.5MB | `firmware/` (自定义 overlay 包) |

## Stage 1: 开启 SSH

**目标**: 检测路由器状态 → 初始化(如需) → 降级(如需) → WiFi 注入开 SSH → 获取密码

```
IP: 192.168.31.1 (stock 固件)
加密: SHA1 (newEncryptMode=0)
```

### Step 1.1: 检测

```bash
bash check_init.sh
# {"init":1} → 出厂未初始化，继续 Step 1.2
# {"init":0} → 已初始化，跳到 Step 1.3
# {"init":-1} → 路由器不可达
```

### Step 1.2: 出厂初始化（仅 init=1 时）

```bash
python3 auto_init.py --ip 192.168.31.1 --ssid Xiaomi_AX6 --wifi-pwd 12345678 --admin-pwd 12345678
# 返回: {"stok":"xxx", "ip":"192.168.31.1", "ssid":"Xiaomi_AX6"}
```

> **⚠️ auto_init 改了 admin 密码后 stok 失效！** auto_init 先用 `admin` 登录拿到 stok，然后改了密码。虽然返回的 stok 可能短时间内有效，但**不可依赖**。

### Step 1.3: 登录获取 stok

```bash
python3 login_get_stok.py --ip 192.168.31.1 --pwd 12345678
# 返回: {"stok":"xxx"}
```

> **必须**: auto_init 之后**必须**重新执行 `login_get_stok.py` 获取新密码下的有效 stok。这是最常被忽略的坑。

### Step 1.4: 检查版本并降级（如需）

```bash
bash check_version.sh
# 如果版本不是 1.0.x (例如 1.1.10 或更高)，需要降级到 1.0.16

python3 downgrade.py --stok STOK --fw firmware/RA69_1.0.16.bin
# 路由器将自动重启，等待约 120 秒重新上线
# 降级后必须重新 login_get_stok.py 获取新 stok
```

### Step 1.5: WiFi 注入开 SSH

```bash
python3 enable_ssh.py --ip 192.168.31.1 --stok STOK --aux-ssid AX6-EXPLOIT --aux-pwd 12345678
```

**攻击链**:
1. `extendwifi_connect` → AX6 连接辅助 WiFi `AX6-EXPLOIT`
2. `oneclick_get_remote_token` → AX6 回调 `169.254.31.1/api/xqsystem/token`
3. 辅助路由器的 exploit_server 返回恶意 token（shell 命令注入）
4. AX6 `eval` 执行 → nvram 开启 SSH → Dropbear 启动

### Step 1.6: 等待 SSH 就绪 + 获取密码

```bash
# 等待端口 22 就绪（通常需要 3-10 秒）
# 检测方法：socket connect 192.168.31.1:22

python3 get_wifi_password.py --ip 192.168.31.1 --stok STOK
# 返回: {"band":"5G", "password":"cc4f0e87"}
```

> **SSH 密码 = 5GHz WiFi 密码**（动态生成，每台不同）。不是固定 `root`。
> 
> **SSH 连接必须指定 HostKeyAlgorithms=+ssh-rsa**。Stock 固件的 Dropbear 只提供 ssh-rsa host key。

### Stage 1 正确命令顺序

```
check_init.sh
  ├─ init=1 → auto_init.py → login_get_stok.py (用新密码 12345678)
  └─ init=0 → login_get_stok.py (用密码 12345678)
check_version.sh
  └─ 非 1.0.x → downgrade.py → 等待重启 → login_get_stok.py
enable_ssh.py --stok STOK
_wait_ssh (socket connect :22 轮询)
get_wifi_password.py --stok STOK
```

## Stage 2: 刷过渡 OpenWRT 到备胎分区

**目标**: 将过渡 OpenWRT 写入空闲的 rootfs 分区，切换启动标志后重启。

**IP**: 192.168.31.1 (stock 固件)  
**方法**: mtd write（**不是 ubiformat**）

### Step 2.1: 确认当前启动分区

```bash
ssh root@192.168.31.1 'cat /proc/cmdline'
# ubi.mtd=rootfs → 当前在 mtd12，备胎是 mtd13 (rootfs_1)
# ubi.mtd=rootfs_1 → 当前在 mtd13，备胎是 mtd12 (rootfs)
```

### Step 2.2: 设置 nvram

无论当前启动分区是什么，统一设置：

```bash
ssh root@192.168.31.1 "
nvram set flag_last_success=0
nvram set flag_boot_rootfs=0
nvram set flag_boot_success=1
nvram set flag_try_sys1_failed=0
nvram set flag_try_sys2_failed=0
nvram set boot_wait=on
nvram set uart_en=1
nvram set telnet_en=1
nvram set ssh_en=1
nvram commit
"
```

> `flag_boot_rootfs=0` 意味着下次启动从 rootfs (mtd12) 启动。这决定了 Stage 2 应该刷入 mtd12。

### Step 2.3: 上传并刷写 xiaomimtd12.bin

```bash
# 上传
scp -O files/openwrt/xiaomimtd12.bin root@192.168.31.1:/tmp/

# 刷写到 mtd12 (rootfs)
ssh root@192.168.31.1 "
mtd erase /dev/mtd12
mtd write /tmp/xiaomimtd12.bin /dev/mtd12
sync
"
```

> **⚠️ 关键**: 使用 `mtd write` 直接写入原始 mtd 镜像。**不要用 ubiformat**，xiaomimtd12.bin 不是 UBI 格式。这是和 AX3600（用 mtd write R3600_mtd12.bin）一致的做法。

### Step 2.4: 重启进入过渡 OpenWRT

```bash
ssh root@192.168.31.1 'reboot'
```

重启后 IP 变为 **192.168.1.1**，SSH **免密登录**。

## Stage 3: 刷 MIBIB + Uboot

**目标**: 在过渡 OpenWRT 中刷入自定义分区表和 u-boot。

**IP**: 192.168.1.1 (过渡 OpenWRT，免密 SSH)

### Step 3.1: 上传文件

```bash
scp -O files/openwrt/ax6-mibib-stock.bin root@192.168.1.1:/tmp/
scp -O files/openwrt/ax6-uboot-stock.bin root@192.168.1.1:/tmp/
```

### Step 3.2: 刷写

```bash
ssh root@192.168.1.1 "
mtd erase /dev/mtd1
mtd write /tmp/ax6-mibib-stock.bin /dev/mtd1
mtd erase /dev/mtd7
mtd write /tmp/ax6-uboot-stock.bin /dev/mtd7
sync
"
```

| 分区 | 设备 | 大小 | 文件 |
|------|------|------|------|
| MIBIB (分区表) | `/dev/mtd1` | 1MB | `ax6-mibib-stock.bin` (256KB) |
| APPSBL (uboot) | `/dev/mtd7` | 1MB | `ax6-uboot-stock.bin` (654KB) |

### Step 3.3: 重启进入新 uboot

```bash
ssh root@192.168.1.1 'reboot'
```

重启后路由器自动进入新 uboot 模式：
1. **蓝灯亮** → 松开 Reset（否则继续按住）
2. uboot 的 HTTPD 在 `http://192.168.1.1/` 就绪

> **LED 灯序**: 上电后等蓝灯亮即可松手。AX6 不需要 AX3600 的「黄→蓝→黄」复杂序列。

## Stage 4: Uboot Web 上传最终固件

**目标**: 通过 uboot web UI 上传 factory.ubi 格式的 ImmortalWRT 固件。

**IP**: 192.168.1.1 (uboot web UI)

### Step 4.1: 确认 uboot 已启动

```bash
curl -s http://192.168.1.1/
# 应返回 HTML 页面，包含 "FIRMWARE UPDATE" 表单
# 底部显示 U-Boot 2016.01 版本信息
```

### Step 4.2: 上传 factory.ubi

```bash
curl -F "firmware=@firmware/immortalwrt-24.10.6-*-factory.ubi" http://192.168.1.1/
```

> **⚠️ 必须是 factory.ubi 格式，不是 sysupgrade.bin。** uboot 不认识 sysupgrade。如果用错了格式会刷不进去。

uboot 返回 "UPDATE IN PROGRESS" 后自动刷写并重启，不需手动干预。

### Step 4.3: 等待系统上线

重启后 IP 仍为 **192.168.1.1**，SSH 免密。

---

## Stage 5: 刷入自定义 Overlay

**目标**: 上传 overlay 包并解压到 /overlay，重启生效。

**IP**: 192.168.1.1 (ImmortalWRT，免密 SSH)

```bash
# 上传
scp -O firmware/overlay-new.tar.gz root@192.168.1.1:/tmp/

# 解压并写入
ssh root@192.168.1.1 "
cd /tmp
tar -xzf overlay-new.tar.gz
cp -a overlay/* /overlay/upper/
rm -rf overlay overlay-new.tar.gz
reboot
"
```

Overlay 包含 `etc/`, `lib/`, `usr/`, `www/` 等自定义配置。重启后生效。

## SSH/SCP 连接要点

| 阶段 | IP | SSH 密码 | Host Key | SCP 参数 |
|------|----|----------|----------|----------|
| Stage 1-2 (stock) | 192.168.31.1 | 5GHz WiFi 密码 | RSA (`-oHostKeyAlgorithms=+ssh-rsa`) | `scp -O` (旧版 Dropbear) |
| Stage 3 (过渡) | 192.168.1.1 | 免密 | RSA | `scp -O` |
| Stage 4-5 (ImmortalWRT) | 192.168.1.1 | 免密 | **ED25519** | 标准 scp |

> **scp 必须加 `-O` 参数**。Stock 和过渡 OpenWRT 的 Dropbear 不支持新版 SFTP 协议。

## 完整 mtd 分区表

```
mtd0:  0:SBL1        (1MB)
mtd1:  0:MIBIB       (1MB)  ← Stage 3 刷分区表
mtd2:  0:QSEE        (3MB)
mtd3:  0:DEVCFG      (512KB)
mtd4:  0:RPM         (512KB)
mtd5:  0:CDT         (512KB)
mtd6:  0:APPSBLENV   (512KB)
mtd7:  0:APPSBL      (1MB)  ← Stage 3 刷 uboot
mtd8:  0:ART         (512KB)
mtd9:  bdata         (512KB)
mtd10: crash          (512KB)
mtd11: crash_syslog   (512KB)
mtd12: rootfs         (35.75MB)  ← Stage 2 刷过渡 OpenWRT
mtd13: rootfs_1       (35.75MB)  ← 双分区备胎
mtd14: overlay        (30.75MB)
mtd15: rsvd0          (512KB)
```

## 完整命令顺序（Copy-Paste 参考）

以下是从头到尾的完整操作序列，IP 和密码根据实际情况调整：

```bash
# === Stage 1: 开 SSH ===
cd AX6
bash check_init.sh                                                    # 检测状态
python3 auto_init.py --admin-pwd 12345678 --ssid Xiaomi_AX6 --wifi-pwd 12345678  # 仅 init=1
python3 login_get_stok.py --pwd 12345678                              # 必须重新登录！
bash check_version.sh                                                 # 检查版本
python3 downgrade.py --stok $STOK --fw firmware/RA69_1.0.16.bin       # 仅非 1.0.x
# 等待路由器重启后重新登录
python3 login_get_stok.py --pwd 12345678
python3 enable_ssh.py --stok $STOK --aux-ssid AX6-EXPLOIT --aux-pwd 12345678
# 等待 SSH 端口 22 就绪
python3 get_wifi_password.py --stok $STOK                             # 获取 SSH 密码

# === Stage 2: 刷过渡 OpenWRT ===
ssh root@192.168.31.1 "nvram set flag_boot_rootfs=0 && nvram commit"  # 下次从 rootfs 启动
scp -O files/openwrt/xiaomimtd12.bin root@192.168.31.1:/tmp/
ssh root@192.168.31.1 "mtd erase /dev/mtd12 && mtd write /tmp/xiaomimtd12.bin /dev/mtd12 && sync"
ssh root@192.168.31.1 reboot
# 等待 192.168.1.1 上线（约 30-60 秒）

# === Stage 3: 刷 MIBIB + Uboot ===
scp -O files/openwrt/ax6-mibib-stock.bin root@192.168.1.1:/tmp/
scp -O files/openwrt/ax6-uboot-stock.bin root@192.168.1.1:/tmp/
ssh root@192.168.1.1 "mtd erase /dev/mtd1 && mtd write /tmp/ax6-mibib-stock.bin /dev/mtd1"
ssh root@192.168.1.1 "mtd erase /dev/mtd7 && mtd write /tmp/ax6-uboot-stock.bin /dev/mtd7 && sync"
ssh root@192.168.1.1 reboot
# 路由器进入 uboot 模式（蓝灯亮）

# === Stage 4: Uboot 上传最终固件 ===
curl -F "firmware=@firmware/immortalwrt-*-factory.ubi" http://192.168.1.1/
# 等待自动刷写 + 重启（约 2-5 分钟）

# === Stage 5: 刷 Overlay ===
scp -O firmware/overlay-new.tar.gz root@192.168.1.1:/tmp/
ssh root@192.168.1.1 "cd /tmp && tar -xzf overlay-new.tar.gz && cp -a overlay/* /overlay/upper/ && rm -rf overlay overlay-new.tar.gz && reboot"
```

## 常见变砖原因 & 避坑指南

| 坑 | 正确做法 | 错误做法 |
|----|----------|----------|
| **auto_init 后 stok 失效** | auto_init 后**必须**重新 `login_get_stok.py` | 直接复用 auto_init 返回的 stok |
| **Stage 2 用了 ubiformat** | 用 `mtd write` 写入 xiaomimtd12.bin | 用 `ubiformat` 写 `.ubi` 文件（文件格式不匹配） |
| **刷到错误分区** | 先看 `/proc/cmdline` 确认当前分区，再写备胎 | 不看分区直接写（可能覆盖正在运行的系统） |
| **flag 未设对** | `flag_boot_rootfs=0` 对应 mtd12 (rootfs) | 写入 mtd12 但 flag=1（启动失败回退 stock） |
| **Uboot 用了 sysupgrade** | uboot web 上传 `.factory.ubi` | 上传 `.sysupgrade.bin`（uboot 不认识） |
| **SSH 密码假设为 root** | 用 `get_wifi_password.py` 获取 5GHz 密码 | 硬编码 `root` 或猜测密码 |
| **跨阶段密码丢失** | 全局 persist 或临时文件持久化 SSH 密码 | 每个阶段独立执行，重新让用户输入密码 |
| **忘记设 nvram boot_wait** | Stage 2 前设 `nvram set boot_wait=on` | 跳过 nvram 设置（可能进不去 uboot） |

## TUI 开发注意事项

1. **Stage 1 必须分两步**：初始化+降级 (Stage 1) → 开SSH (Stage 1.5)。因为 auto_init 后需要重新登录，如果合并在一个函数里容易踩坑。

2. **SSH 密码跨阶段持久化**：Stage 1 获取的密码和 stok 必须在 Stage 2 可用。用全局 dict（如 `persist = {"ssh_password": None, "stok": None}`）在进程内传递。

3. **Stage 2 IP 会变**：从 192.168.31.1 变为 192.168.1.1。Stage 2 重启后 TUI 需要等待新 IP 上线。

4. **Stage 3→4 之间需要人工操作**：刷完 uboot 重启后，用户需要确认路由器已进入 uboot 模式。TUI 应在此时提示用户并等待确认，不要全自动跳过。

5. **不要自动关闭 exploit_server**：exploit_server 在辅助路由器上独立运行，TUI 不应管理它的生命周期。

6. **每个 Stage 调对应 CLI 脚本**：不要重复实现脚本逻辑。flasher 类只做编排和异常处理。
