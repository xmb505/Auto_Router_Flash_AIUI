# 项目文档交叉审计

当项目有多层文档（顶层约定 → 机型文档 → 代码实现）且需要确保三者一致时使用。

## 适用场景

- 接手一个已有文档和代码的中等规模项目
- 重构/改名前需要了解当前真实状态
- 多个机型/模块有独立文档，但共享顶层规范
- 发现"某个地方看起来不对劲"，需要系统排查

## 流程

### 第一步：构建文档索引

用 glob 列出三类文件：

1. **顶层约定/规范**：`doc/conventions/*`、`QWEN.md` 等
2. **模块/机型文档**：`src/project/*/doc/*` 等
3. **实际代码**：`src/project/*/*.py`、`src/project/*/*.sh` 等

### 第二步：阅读，带追问视角

读每个文件时追问：

| 视角 | 问题 |
|------|------|
| **规范 vs 规范** | 多份规范是否自洽？引用同一概念时用词/定义是否一致？ |
| **规范 vs 代码** | 规范要求做的事，代码做到了吗（如 `--help-json`、`reason`/`recoverable`）？ |
| **规范 vs 文档** | 规范要求每机型必须有的文档，实际都存在吗？ |
| **代码 vs 代码** | 同功能在不同机型的实现中参数名/签名/行为是否一致？ |
| **文档 vs 文档** | 不同文档对同一事实的描述是否有矛盾（如"已实测"vs"未实测"）？ |
| **旧痕检测** | 规范/文档中是否有已被后续修订否定的残影（如引用了不存在的字段/模块）？ |

### 第三步：归类和标记

把发现的问题分类：

- **接口漂移**：参数名、字段名在不同地方不一致
- **残影条目**：规范提到但已被其他规范否定或实际不存在的特性
- **缺失条目**：规范要求有但实际没有（文档、功能、模块）
- **事实矛盾**：两处对同一事实的描述冲突
- **规范漂移**：同一规范在多个文件中定义但不同版本

### 第四步：聚焦提问

不要一次问所有问题。分组问：

1. 优先级最高的（命名冲突 / 行为矛盾 / 可能引起运行时错误）
2. 中等优先级（文档缺失 / 格式不统一）
3. 低优先级（残影 / 重复内容）

### 第五步：落实修复

对每个确认的问题：

1. 修改**规范**文件统一定义
2. 修改**代码**对齐新定义
3. 修改**机型文档**更正事实

修改完后用 `git diff` 或重新 grep 确认一致性。
